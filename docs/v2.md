## V2 — Monitoramento completo de grupos

### Objetivo

Para fontes `tipo = group`, persistir **todo** tráfego relevante: remetente (JID/número), tipo de mensagem, citações, menções, mídias no MinIO com prefixo por fonte, e metadados brutos suficientes para reconstruir contexto depois.

### Tipos de mensagem (extensível)

Os valores são **strings** no banco (não ENUM SQL), para aceitar novos tipos sem migration:

`sistema sugerido (inicial):` texto, áudio, imagem, vídeo, documento, figurinha, localização, contato, enquete, reação, url, desconhecido, outro.

Novos tipos podem surgir da Evolution/API; gravar como `desconhecido` ou `outro` com `metadata` completo até mapearmos.

### Pipeline de enriquecimento (futuro)

| Ativo | Etapa | Fallback |
|--------|--------|----------|
| Áudio | Transcrição Whisper (local) | NeuronAI / provedor configurado |
| Imagem | (sem OCR local confiável no início) | LLM visão via NeuronAI |
| Texto | Classificação / categorias | já previsto no fluxo AI |

Campos dedicados na tabela de mensagens (`transcription_*`, `vision_*`, status de processamento) permitem retries idempotentes e auditoria **sem** misturar tudo em JSON opaco.

### Armazenamento S3 / MinIO

- Bucket compartilhado (`pessoal`), **prefixo por fonte monitorada** (`monitored_sources.media_storage_prefix`).
- Anexos em tabela própria (`message_attachments`) com `storage_path` = chave no bucket.
- Evolução futura: política de lifecycle por prefixo (não depende do MVP).

## V2 — Dashboard e permissões por grupo

### Personas

- **Dono global (Raphael):** acesso a pessoal + todos os grupos monitorados + configurações sensíveis.
- **Admin de grupo (futuro):** acesso ao dashboard **somente** dados ligados a uma ou mais fontes `group` específicas (ex.: moderar leitura de histórico/mídia daquele grupo).

### Modelo de dados (preparação)

- `users.global_role`: `super_admin` | `member` — define se o usuário pode operar em escopo global (super_admin) ou apenas no que for explicitamente concedido (`member`).
- `monitored_source_user`: vínculo N:N entre usuário e fonte monitorada, com papel no contexto daquela fonte (ex.: `group_admin`, `viewer`).

A **V1** pode ignorar totalmente o pivot: só você existirá como `super_admin` via seeder/manual. A existência da tabela **não quebra** o app.

### Autorização no código (quando implementar)

- Gates/policies checando `global_role === super_admin` OU existência de registro em `monitored_source_user` com papel adequado para a rota recurso.

## Como evoluir sem quebrar a V1

1. **Novas colunas opcionais** com default / nullable — ok.
2. **Novas tabelas** referenciando as existentes — ok; código antigo não as usa.
3. **Evitar** renomear colunas em uso; se inevitável, migration de renomeação + janela de deploy.
4. **Feature flags** (.env) para ativar Whisper, LLM visão, ou UI de “admin de grupo” quando estiver pronto.

---

## Backlog sugerido (ordem não rígida)

1. Webhook estável + persistência completa grupo + anexos no MinIO.
2. Jobs de transcrição (fila dedicada) com retry e marcação de status.
3. Classificação/categorias e “responsável” (humano ou regra) — pode exigir colunas adicionais; propor no SPEC antes.
4. UI admin por grupo + convites (se necessário).
